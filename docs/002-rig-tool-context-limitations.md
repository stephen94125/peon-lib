# Architecture Decision Record: Unresolvable `rig` Tool Context Limitations

## Context and Problem Statement
In the `peon-telegram` application, we require strict Zero-Trust security isolation. When a user interacts with the bot, the user's specific Telegram `chat_id` and `user_id` must be securely passed down to the tool execution layer (e.g., `ExecuteScriptTool`) to ensure that `casbin` validates the specific user's permission to execute a task.

Additionally, to provide a coherent user experience in multi-user group chats, we require a persistent `Agent` session (via a state manager like `moka`) instead of instantiating a fresh, amnesiac agent for every message. 

However, we encountered a fundamental architectural limitation heavily rooted in the underlying `rig` crate. 

## The Core `rig` Limitations

### 1. `tokio::task_local!` Fails Due to Background ToolServer
Initially, we attempted to use Tokio's `task_local!` macro (`CURRENT_UID.scope(...)`) to transparently pass the user ID from the Telegram webhook down to the tools. 
This failed because when `rig::agent::AgentBuilder::build()` is called, it spawns a dedicated background Tokio actor (`ToolServer`) and dispatches tool execution requests over an `mpsc::channel`. This cross-thread dispatching entirely breaks Tokio's task-local scope, causing the UID to drop back to the default `"agent"` identity.

### 2. Missing Context Parameter in `rig::tool::Tool` Signature
The `rig` library's `Tool` trait provides no parameters to pass request-scoped context:
```rust
fn call(&self, args: Self::Args) -> impl Future<Output = Result<Self::Output, Self::Error>>;
```
The tool only receives `&self` (the global tool configuration) and `args` (the JSON payload generated by the LLM). We cannot inject the UID into `args` via system prompts, because malicious users could trick the LLM into generating an escalated privilege UID (hallucination/injection).

### 3. Concurrency Hazards with Interior Mutability
If we combine a long-lived `Agent` (cached per-group) with an interior mutable state (like `tokio::sync::RwLock<String>`) inside `PeonEngine` to store the active UID, we create severe Race Conditions. If two users in the same Telegram group send messages simultaneously, rewriting the shared UID will cause requests to override each other midway through LLM execution, leading to unpredictable security leaks or broken sessions.

While we *could* wrap the entire `Agent` in a `tokio::sync::Mutex`, serializing group-chat concurrency, it represents a substantial architectural code smell to globally mutate core engine properties right before a multi-turn LLM generation step.

## Decision
**This problem is fundamentally unresolvable** while maintaining both high concurrency and zero-trust security under the current `rig` architecture.

We must halt attempts to magically inject contextual user states into `rig` tools. 
Until the authors of the `rig` library publish an update containing an official mechanism for Dependency Injection / `Context` propagation inside `Tool::call`, this issue is recorded as blocked.
